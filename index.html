Introduction

What is Git?

Git is a free and open source distributed version control system designed to handle everything from small to very large projects with speed and efficiency. It helps developers track changes in their code, collaborate with others, and manage different versions of a project.


Instead of keeping multiple copies of files, Git stores snapshots of your project’s history, allowing you to see what changed, when it changed, and who made the changes. It makes teamwork easier by letting multiple people work on the same project simultaneously without overwriting each other’s work. Git also provides tools to create branches for experimenting or adding new features, and later merge them back into the main project once they’re ready.


Key aspects of Git:

  • Version Control: Git allows developers to record every change made to a project's files, creating a complete history of modifications.
  • Distributed Nature: Every developer working with a Git repository has a complete, local copy of the entire project history. This offers advantages such as offline work capabilities, faster operations (like committing and branching), and increased resilience to server failures.
  • Collaboration: Enabling multiple developers to work on the same project simultaneously without overwriting each other's work. It provides robust features for branching, merging, and resolving conflicts, allowing teams to integrate changes efficiently.
  • Performance and Flexibility: Its design emphasizes performance in common operations like committing, branching, and merging, and it supports a wide range of workflows tailored to different team needs.

What is GitHub?

GitHub is a cloud-based platform built on the Git version control system that allows developers to store, manage, track changes in, and collaborate on code and other project files. It provides a central online location for storing projects in repositories, enabling teams to work together, contribute to each other's work, and build software. GitHub also offers tools for project management, bug tracking, and code improvement within a collaborative environment.

Setting Up Git & GitHub

  1. Install Git

    Open up your terminal and run the following commands line-by-line:

    // On Linux:
    sudo apt update
    sudo apt upgrade
    sudo add-apt-repository ppa:git-core/ppa
    sudo apt update
    sudo apt install git
    git --version

    // On MacOS:
    /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
    brew install git
    git --version

  2. Configure Git & GitHub

    1. Create a GitHub Account

    2. Open your terminal and run the following commands with the information you signed up with in GitHub:

      // Run the following commands line-by-line:
      git config --global user.name "Your Name"
      git config --global user.email yourname@example.com
      git config --global init.defaultBranch main
      git config --global pull.rebase false

      // To verify that things are working properly, run the following commands and verify whether they match your name and email:
      git config --get user.name
      git config --get user.email

    3. Create an SSH key

      An SSH key is a cryptographically secure identifier used to identify your machine/device. GitHub uses SSH keys to allow you to upload to your repository without having to type in your username and password every time. Run the following commands in your terminal to set it up:

      // Run this to check if you already have an SSH key installed
      ls ~/.ssh/id_ed25519.pub

      // If a message in the console says "No such file or directory," run this code, else skip this step:
      ssh-keygen -t ed25519 // It will prompt you for a location and password to save the key. Simply hit enter in both cases.

    4. Link your SSH key with GitHub

      Log-in to GitHub -> Click your Profile Picture -> Settings -> SSH and GPG Keys -> New SSH Key

      Name your key as something descriptive.

      Next, copy your public SSH key by running this in the terminal. Highlight and copy the output:

      cat ~/.ssh/id_ed25519.pub

      Go back to your GitHub browser window and paste the key you copied into the key field. Keep the key type as Authentication Key then click Add SSH Key

      Congratulations! You're done!

Git Flow

A Branch in git represents a version or snapshot of your project. In github, this can be versions of an application, specific bug fixes, or newly created features in your project.


A Git Merge or Merging branches is basically taking your branches (different versions/features/fixes) and, you guessed it, merging them together to add/update these new features into your project.

Here's everything you need to know:

  • Main: Your Main/Master branch represents the official version releases of your projects out in the public.
  • Hotfix: Your Hotfix branches are quick bug fixes in case your released Main branch contains bugs.
  • Release: Your Release branch is your "release-testing" branch. Here, you start the release cycle for testing, bug fixes, resolving merge conflicts, and other release operations before officially merging up to main.
  • Development: Your Development branch is where you merge all your completed features. Once enough features are merged, you funnel them up to your Release branch for testing and resolving merge conflicts.
  • Feature: Your feature branches are code/features you want to introduce into your project. You can create multiple of them (for multiple features), and once completed, they will typically funnel up to your development branch.

Git Methods

Here is a list of basic/common Git commands you should know to work on your project:


  • Stage & Snapshot

    // 1. Shows current branch and modified files in working directory
    git status

    // 2. Adds file/s to stage
    git add [file/s]

    // 3. Unstage a file/s
    git reset [file/s]

    // 4. Difference of what is changed but not staged
    git diff

    // 5. Difference of what is staged but not committed
    git diff --staged

    // 6. Commit staged files
    git commit -m "[descriptive message]"

  • Branch & Merge

    // 1. Lists all your branches
    git branch

    // 2. Create a new branch with branch name
    git branch [branch-name]

    // 3. Switch to another branch
    git checkout [branch-name]

    // 4. Merge the specified branch into the current one
    git merge [branch]

  • Inspect & Compare

    // 1. Shows commit history of current branch
    git log

    // 2. Shows commits on BranchA not in BranchB
    git log branchB..branchA

    // 3. Shows diff of what's in BranchA not in BranchB
    git diff branchB...branchA

  • Share & Update

    // 1. Clones remote repository into local repository (perform once)
    git clone [repository-url]

    // 2. Fetches remote repository into local repository (not merged)
    git fetch [remote-repo-alias e.g. origin]

    // 3. Merge remote branch to current branch
    git merge [remote-repo-alias e.g. origin]/[branch-name]

    // 4. Fetch & Merge commits from tracking remote branch
    git pull

    // 5. Merge the specified branch into the current one
    git push [remote-repo-alias e.g. origin] [branch e.g. main]

  • Undo or Revert Changes/Commits

    // Reverts to a commit (A->B->C ===> A->B->C->B')
    git revert [commit-hash]

    // Reset to a commit (A->B->C ===> A->B)
    // Note: soft (none), mixed(resets stage), hard(resets stage and files)
    git reset --soft/mixed/hard [commit-hash]

    // Restore file/s to a previous commit
    git restore [file/s]

Git Strategies

Here are two basic strategies you can use when working with Git that you can use today! They were created by Sam Livingston-Gray in this higly recommended read to Think Like (a) Git


  • The Scout Pattern

    Here, you create a new branch off of main as a scout/test branch. You merge your features into the test branch, then merge the test branch to main if you're happy with the result.

    Overview:

    1. Make sure you're in the right branch with a clean, working slate.
    2. Create a new branch (test_branch). Switch to it.
    3. Do the merge (feature_branch -> test_branch).
    4. Are you happy with the result?
      • Yes: Merge test_branch -> main_branch
      • No: Delete test_branch
      // 1. Must be in main branch!
      git status

      // 2. Create and go to new branch.
      git checkout -b test_branch

      // 3. Merge feature branch to test branch
      git merge feature_branch

      // 4. Are you happy?
      // Yes, I am!
      git checkout main
      git merge test_branch

      // No, I'm not!
      git checkout master
      git branch -D test_merge

  • The Savepoint Pattern

    Here, you create a new branch off of main as a savepoint. You directly merge your features into main, and revert back to your savepoint if you aren't happy with the result.

    Overview:

    1. Make sure you're in the right branch with a clean, working slate.
    2. Create a new branch (save_branch). Don't switch to it.
    3. Do the merge (feature_branch -> main_branch).
    4. Are you happy with the result?
      • Yes: Delete save_branch.
      • No: Reset main_branch to save_branch
      // 1. Must be in main branch!
      git status

      // 2. Create new branch.
      git branch save_branch

      // 3. Merge feature branch to main branch
      git merge feature_branch

      // 4. Are you happy?
      // Yes, I am!
      git branch -d save_branch

      // No, I'm not!
      git reset --hard save_branch
